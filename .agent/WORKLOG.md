# 作業ログ (Work Log) 運用ガイド

## 目的

本ドキュメントは、実行計画の実行時に発生した課題/問題/検証結果を「ログ」として
保存し、後からチームミーティング向けの報告書へまとめ直せるようにするための運用を
定義するものである。

## 置き場所

- 作業ログ: `.agent/worklog/<YYYY-MM-DD>_<topic>.md`
- 報告書（生成物）: `.agent/reports/<YYYY-MM-DD>_<topic>.md`（任意）

## どの粒度で書くか

作業ログは「細かく」残してよいが、後で粒度調整できるように 1 エントリーを
最小単位として書く。

- 1 行（1 箇条書き）= 1 エントリーとする。
- 追加の補足が必要な場合は、同じ箇条書きの下にインデントした行で追記する。
  （同じ論点を維持する）

## タグ（推奨）

後から抽出しやすくするため、エントリーにタグを付ける。

- `#issue`: 課題/問題（未解決、ブロッカー、原因調査が必要）
- `#decision`: 意思決定（選択、採用、却下、その理由）
- `#risk`: リスク（回帰、運用、性能、互換性）
- `#verify`: 検証結果（再現条件、コマンド、期待と観測、ログ要点）
- `#todo`: 次アクション（具体的で、依存と完了条件が明確）
- `#note`: 参考情報（リンク、メモ、背景）

## 推奨フォーマット

日付見出しの下に、時刻付きで箇条書きする。

```md
# Title: 2026-02-05_deimv2_hailo8_port の作業ログ

## 2026-02-11

- 10:32 #issue HailoRT の scheduler 有効時に `activate()` が失敗する条件を再現した。
  - 期待: 手動 activate で network group を有効化できる
  - 観測: `HAILO_INVALID_OPERATION`
- 11:10 #decision vstream 経路のみで推論し、手動 activate は呼ばない方針とした。理由: scheduler と競合するため。
- 14:05 #verify 実機で `ctest -R '^DNN\\.HailoTest$'` を再実行し、4 variant で通過を確認した。
```

## 報告書へのまとめ方（例）

ミーティング向け報告は、以下の順でまとめると議論が進みやすい。

1. `#issue`（未解決とブロッカー）
2. `#decision`（意思決定と理由）
3. `#risk`（今週のリスクと緩和策）
4. `#verify`（重要な検証結果の要点）
5. `#todo`（次アクションと期限）

作業ログが構造化されていれば、タグ単位で抽出して粒度を上げ下げできる。
